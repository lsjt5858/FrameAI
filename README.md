# FrameAI

FrameAI 是一个面向 AI 漫剧 / AI 视频生产流程的内部工作流中台，目标不是做一个单纯的 API 聚合站，而是把项目、素材、提示词、分镜、生成任务和结果管理串成一个可复用、可追踪、可扩展的生产闭环。

当前仓库已经进入 MVP 开发阶段，核心设计依据见 `开发指南.md`，开发进度跟踪见 `docs/PROGRESS.md`。

## 项目定位

FrameAI 聚焦于 AI 视频制作中最耗时、最容易断链的环节：

- 项目管理
- 素材管理
- 提示词模板复用
- 生图任务调度
- 生视频任务调度
- 任务状态跟踪
- 结果对比与入选
- 素材来源追踪
- API 调用日志与成本记录

它的核心价值不是“接了多少平台”，而是让同一个角色、同一套参考图、同一套提示词模板，能够被批量生成、对比、复用和追踪。

## 产品目标

第一阶段定位为个人内部工具，优先服务真实的 AI 视频制作工作流，而不是一开始就做成商业 SaaS。

目标是跑通下面这条最小闭环：

`创建项目 -> 上传参考图 -> 编写/套用提示词 -> 生图 -> 选图 -> 生视频 -> 对比结果 -> 保存入库 -> 后续复用`

## 解决的问题

AI 视频制作真正的难点通常不在页面搭建，而在工作流和数据链路：

- 不同平台 API 规则不一致，有同步、异步、轮询、回调等多种方式
- 生视频任务耗时长，必须支持任务状态管理、失败重试和后台执行
- 素材链路复杂，角色图、场景图、分镜图、视频结果都需要追踪来源
- API 成本难控制，需要记录模型、耗时、额度、错误和成功率
- 提示词与参数需要沉淀成模板，才能真正提升复用效率

## MVP 范围

第一版只做最小可用闭环，不追求大而全。

### 做什么

- 项目管理
- 素材库
- 提示词模板库
- 接入 1 个生图平台
- 接入 1 个生视频平台
- 基础任务状态管理：`pending / running / succeeded / failed`

### 暂时不做什么

- 多用户登录
- 团队权限
- 商业计费
- 自动成片
- 复杂剪辑
- 多平台同时调度

## 核心模块

### 1. 项目管理

每个视频或漫剧作为一个项目，统一承载角色、场景、分镜、素材和生成结果。

### 2. 素材库

管理角色图、服装图、场景图、参考图、成片视频等素材，并记录：

- 来源
- 所属项目 / 分镜
- 使用的平台和模型
- 原始提示词
- 关键参数
- 上游参考图
- 是否入选

### 3. 提示词模板库

沉淀高频模板，提升批量生产效率，例如：

- 角色三视图模板
- 服装细节模板
- 场景四宫格模板
- 人物一致性模板
- 图生视频模板
- 首尾帧视频模板

### 4. 生图中心

统一选择模型、比例、清晰度、参考图、提示词模板和批量数量，并调用统一适配层完成任务创建。

### 5. 生视频中心

支持图生视频、首尾帧视频、文生视频，重点关注：

- 任务状态
- 失败重试
- 批量生成
- 结果对比
- 一键保存到素材库

### 6. 分镜工作台

分镜工作台是 AI 漫剧场景中最核心的页面。每个镜头可以承载：

- 镜头编号
- 剧情描述
- 角色
- 场景
- 参考图
- 生图提示词
- 生视频提示词
- 生图结果
- 生视频结果
- 是否通过
- 备注

### 7. API 适配层

业务层不直接调用第三方平台 API，而是通过统一接口访问：

```text
generateImage()
generateVideo()
getTaskStatus()
downloadResult()
```

后续每个平台独立实现 adapter，避免业务逻辑与平台细节耦合。

### 8. 任务队列与状态管理

所有生图 / 生视频任务都进入后台任务体系，而不是由前端长时间阻塞等待。

建议的状态流转：

```text
pending
running
succeeded
failed
cancelled
```

### 9. 成本与日志

记录每次调用的关键信息：

- 使用平台
- 使用模型
- 输入参数
- 开始时间 / 结束时间
- 执行耗时
- 成功 / 失败
- 错误信息
- 预估消耗
- 输出结果

## 用户工作流

一个典型的使用流程如下：

1. 创建项目
2. 创建角色、场景与分镜
3. 上传角色图、场景图和参考图
4. 在分镜或任务页套用提示词模板
5. 发起生图任务并筛选结果
6. 基于入选图继续发起生视频任务
7. 对比视频结果并标记通过
8. 将最终结果保存回素材库，保留来源链路

## 技术方案

第一版建议采用轻量、可快速迭代的技术栈：

- 前端：React
- 后端：FastAPI
- 数据库：SQLite，后续可迁移到 PostgreSQL
- 文件存储：本地 `storage/`
- 队列：先使用数据库任务表 + 后台轮询，后续升级到 Redis / Celery

这个组合适合先把流程跑通，再逐步扩展到更复杂的部署方式。

## 数据模型草案

核心表建议包括：

```text
projects
characters
scenes
shots
assets
prompt_templates
generation_tasks
api_call_logs
```

关键关系：

- 一个 `project` 拥有多个 `shot`
- 一个 `shot` 可以绑定多个 `asset`
- 一个 `generation_task` 可以生成一个或多个 `asset`
- 一个 `asset` 可以继续作为后续任务的参考输入

## 页面规划

第一版建议包含以下页面：

1. 首页 / 项目列表
2. 项目详情
3. 分镜工作台
4. 素材库
5. 提示词模板库
6. 生图任务页
7. 生视频任务页
8. 任务记录页
9. 设置页（API Key、默认模型、存储路径）

## 实现顺序

推荐按下面顺序推进，优先保障闭环：

1. 搭建项目骨架：FastAPI + React + SQLite
2. 实现数据库模型与本地文件上传
3. 完成项目管理与素材库
4. 完成提示词模板功能
5. 接入第一个生图 API adapter
6. 接入第一个生视频 API adapter
7. 实现任务状态查询与轮询
8. 实现分镜工作台
9. 实现结果对比与入选标记
10. 实现日志、成本统计与失败重试
11. 再扩展更多平台

## 阶段路线图

### Phase 1: MVP 闭环

目标：在一个项目内跑通生图和生视频的最小工作流。

### Phase 2: 分镜工作台

目标：让平台真正贴合 AI 漫剧生产方式。

### Phase 3: API 适配层

目标：后续接入更多平台时，不改业务主流程。

### Phase 4: 素材链路追踪

目标：明确每个素材的来源、参数和上下游关系。

### Phase 5: 批量生成与结果对比

目标：提升复用效率和筛选效率。

### Phase 6: 成本与日志分析

目标：控制 API 消耗并形成稳定复盘能力。

## 验收标准

第一版完成时，至少应该满足：

- 能创建一个 AI 视频项目
- 能上传角色图、场景图和参考图
- 能创建分镜并填写提示词
- 能调用一个生图 API 并保存结果
- 能基于生图结果调用一个生视频 API
- 能查看任务状态和失败原因
- 能把满意结果标记为入选
- 能回看每个素材的来源、提示词和参数

## 本地运行

后端：

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

前端：

```bash
cd frontend
npm install
npm run dev
```

默认访问：

- 前端工作台：http://localhost:5173
- 后端 API：http://localhost:8000
- API 文档：http://localhost:8000/docs

## Provider 配置

默认使用 `mock` provider，不需要任何 API Key，适合本地验证任务、素材、分镜和日志链路。

真实生图可配置 OpenAI：

```bash
OPENAI_API_KEY=sk-...
OPENAI_IMAGE_MODEL=gpt-image-1
```

真实生视频可配置 DashScope Wan：

```bash
DASHSCOPE_API_KEY=...
DASHSCOPE_REGION=intl
DASHSCOPE_VIDEO_MODEL=wan2.6-i2v
```

注意：云端图生视频通常需要公网可访问的图片 URL。本地 `storage/` 文件不能直接被外部平台拉取，如需图生视频或首尾帧视频，需要配置：

```bash
FRAMEAI_PUBLIC_STORAGE_BASE_URL=https://your-domain.example/storage
```
