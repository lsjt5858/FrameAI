好实现，但建议别一开始做成“大而全平台”。更现实的定位是：**AI 视频制作工作流中台**，先把“生图、生视频、素材管理、提示词模板、任务队列、结果对比”跑通，再逐步接更多平台 API。

难点不在前端页面，而在这些地方：

- 各平台 API 规则不同：有的是同步返回，有的是异步任务，有的要轮询，有的有回调。
- 生视频任务耗时长：必须做任务队列、状态管理、失败重试。
- 素材链路复杂：参考图、角色图、场景图、分镜图、视频结果都要能追踪来源。
- 额度和成本控制：每次调用哪个模型、花了多少钱、失败了几次，都要记录。
- 提示词和参数复用：真正提升效率的是模板化和批量化，而不是只做一个“调用按钮”。

可以按这个大纲做。

**平台定位**
做一个内部 AI 漫剧/AI 视频生产平台，统一管理：

- 项目
- 剧本
- 角色
- 场景
- 分镜
- 参考图
- 生图任务
- 生视频任务
- 提示词模板
- API 调用记录
- 成品素材

**核心模块**
1. 项目管理  
每个视频/漫剧一个项目，里面放角色、场景、分镜、生成结果。

2. 素材库  
管理角色图、服装图、场景图、参考图、成片视频。每个素材记录来源、使用过的模型、提示词和参数。

3. 提示词模板库  
把常用模板沉淀下来，比如：

```text
角色三视图模板
服装细节模板
场景四宫格模板
人物一致性模板
图生视频模板
首尾帧视频模板
达芬奇调色备注模板
```

4. 生图中心  
统一接入多个生图平台 API。页面上只选择：

- 模型
- 比例
- 清晰度
- 参考图
- 提示词模板
- 批量数量

5. 生视频中心  
支持图生视频、首尾帧视频、文生视频。重点要有：

- 任务状态
- 失败重试
- 批量生成
- 结果对比
- 一键保存到项目素材库

6. 分镜工作台  
这是最适合 AI 漫剧的核心页面。

每个镜头包含：

- 镜头编号
- 剧情描述
- 角色
- 场景
- 参考图
- 生图提示词
- 生视频提示词
- 生成结果
- 是否通过

7. API 适配层  
不要让业务代码直接调用各平台 API。要做统一接口，例如：

```text
generateImage()
generateVideo()
getTaskStatus()
downloadResult()
```

然后每个平台单独写 adapter。

8. 任务队列  
生图/生视频都应该进入任务队列，而不是前端一直等。

状态可以是：

```text
pending
running
succeeded
failed
cancelled
```

9. 成本和日志  
记录每次调用：

- 使用平台
- 使用模型
- 输入参数
- 任务耗时
- 成功/失败
- 消耗额度
- 生成结果

**推荐 MVP**
第一版不要接太多平台，先做这 5 个功能：

1. 项目管理
2. 素材库
3. 提示词模板库
4. 接一个生图 API
5. 接一个生视频 API

等这个流程跑通后，再加批量生成、分镜工作台、多平台切换。

**技术架构**
比较适合：

- 前端：Next.js / React
- 后端：Node.js 或 Python FastAPI
- 数据库：PostgreSQL
- 文件存储：本地、S3、Cloudflare R2 或阿里云 OSS
- 队列：Redis + BullMQ，或者 Celery
- 登录：先做本地账号，后续再加团队权限

**最终形态**
你真正需要的不是“API 聚合平台”，而是：

> 一个围绕 AI 视频生产的素材、提示词、分镜、任务和结果管理系统。

API 聚合只是底层能力。真正让你效率提升的是：**同一个角色、同一套参考图、同一套提示词模板，可以批量生成、对比、复用、追踪。**

建议第一阶段就做成“内部工具”，不要急着做商业 SaaS。先服务你自己的 AI 漫剧生产流程，等流程稳定了，再考虑开放给别人用。





# AI 视频制作工作流中台开发大纲

## Summary
第一阶段做一个“个人内部工具”，不是商业 SaaS。目标是把 AI 漫剧/AI 视频制作中最费时间的环节统一起来：项目、角色、场景、分镜、参考图、提示词、生图、生视频、任务状态、结果对比、素材复用。

第一版技术方案：
- 前端：React
- 后端：FastAPI
- 数据库：SQLite，后续可迁移 PostgreSQL
- 文件存储：本地 `storage/`
- 队列：先用数据库任务表 + 后台轮询，后续再上 Redis/Celery
- 首批 API：1 个生图平台 + 1 个生视频平台

## 开发大纲写法
开发文档建议拆成 6 部分：

1. 产品目标  
说明这个平台解决什么问题：提升 AI 视频制作效率，不是单纯聚合 API。

2. 用户流程  
写清楚你每天怎么用它：创建项目 -> 写分镜 -> 上传参考图 -> 套提示词 -> 生图 -> 选图 -> 生视频 -> 对比结果 -> 入库复用。

3. 功能模块  
按模块拆：项目管理、素材库、提示词模板、分镜工作台、生图中心、生视频中心、任务队列、日志成本。

4. 数据结构  
定义项目、角色、场景、分镜、素材、任务、提示词模板这些核心对象。

5. API 接口  
定义前端调用后端的接口，以及后端调用外部平台的统一 adapter。

6. 里程碑  
不要一次做完。按 MVP -> 工作流闭环 -> 批量生成 -> 多平台扩展逐步实现。

## Phase 1: MVP 闭环
目标：先跑通“一个项目里完成生图和生视频”的最小闭环。

实现内容：
- 项目管理：创建、编辑、删除项目。
- 素材库：上传图片/视频，查看素材，记录素材来源。
- 提示词模板：创建模板、编辑模板、套用模板。
- 生图任务：输入提示词和参数，调用一个生图 API，保存结果。
- 生视频任务：选择参考图，输入视频提示词，调用一个生视频 API，保存结果。
- 任务列表：展示 `pending / running / succeeded / failed`。

第一阶段不做：
- 多用户登录
- 商业计费
- 团队权限
- 复杂剪辑
- 自动成片
- 多平台同时调度

## Phase 2: 分镜工作台
目标：让平台真正贴合 AI 漫剧制作。

每个分镜包含：
- 镜头编号
- 剧情描述
- 角色
- 场景
- 参考图
- 生图提示词
- 生视频提示词
- 生图结果
- 生视频结果
- 是否通过
- 备注

核心操作：
- 为一个分镜绑定角色图、服装图、场景图。
- 一键套用“角色三视图 / 场景四宫格 / 图生视频”模板。
- 一键生成多张图。
- 从结果里标记“入选图”。
- 用入选图继续生成视频。
- 把入选视频标记为“通过”。

## Phase 3: API 适配层
目标：后续接更多平台时，不改业务逻辑。

统一接口设计：
```text
generateImage(provider, prompt, referenceImages, params)
generateVideo(provider, prompt, referenceImages, params)
getTaskStatus(provider, taskId)
downloadResult(provider, resultUrl)
```

每个平台单独实现 adapter：
```text
OpenAIImageAdapter
WanxiangVideoAdapter
KlingVideoAdapter
VolcengineAdapter
ReplicateAdapter
```

业务层只关心：
```text
我要生图
我要生视频
我要查任务状态
我要下载结果
```

不直接关心每个平台 API 的细节。

## Phase 4: 素材链路追踪
目标：知道每个素材是怎么来的，方便复用和复盘。

每个素材记录：
- 文件路径
- 类型：图片 / 视频 / 音频 / 文档
- 所属项目
- 所属分镜
- 来源：上传 / 生图 / 生视频 / 外部导入
- 使用的平台
- 使用的模型
- 原始提示词
- 参数
- 上游参考图
- 生成时间
- 是否入选

这样可以追踪：
- 这张角色图来自哪个提示词。
- 这个视频用了哪张参考图。
- 哪个模型最稳定。
- 哪套提示词最适合复用。

## Phase 5: 批量生成和结果对比
目标：真正提升效率。

功能：
- 同一个提示词批量生成多张图。
- 同一个参考图批量生成多个视频。
- 多结果网格对比。
- 标记喜欢、不喜欢、废弃。
- 支持复制某次任务参数重新生成。
- 支持把成功任务保存为模板。

## Phase 6: 成本和日志
目标：控制 API 消耗。

记录每次调用：
- 平台
- 模型
- 输入参数
- 开始时间
- 结束时间
- 状态
- 错误信息
- 预估消耗
- 输出文件

页面展示：
- 今日调用次数
- 本项目调用次数
- 成功率
- 失败任务
- 最常用模型
- 最常用模板

## 数据模型草案
核心表：

```text
projects
characters
scenes
shots
assets
prompt_templates
generation_tasks
api_call_logs
```

最重要的关系：
- 一个 `project` 有多个 `shot`
- 一个 `shot` 可以绑定多个 `asset`
- 一个 `generation_task` 会生成一个或多个 `asset`
- 一个 `asset` 可以继续作为后续任务的参考图

## 页面大纲
第一版页面：

1. 首页 / 项目列表
2. 项目详情
3. 分镜工作台
4. 素材库
5. 提示词模板库
6. 生图任务页
7. 生视频任务页
8. 任务记录页
9. 设置页：API Key、默认模型、存储路径

## 推荐实现顺序
1. 搭项目骨架：FastAPI + React + SQLite。
2. 做数据库模型和本地文件上传。
3. 做项目管理和素材库。
4. 做提示词模板。
5. 做生图 API adapter。
6. 做生视频 API adapter。
7. 做任务状态和轮询。
8. 做分镜工作台。
9. 做结果对比和入选标记。
10. 做日志、成本、失败重试。
11. 再接更多平台。

## Acceptance Criteria
第一版完成标准：

- 能创建一个 AI 视频项目。
- 能上传角色图、场景图、参考图。
- 能创建分镜并填写提示词。
- 能调用一个生图 API 并保存结果。
- 能用生图结果调用一个生视频 API。
- 能看到任务状态和失败原因。
- 能把满意结果标记为入选。
- 能回看每个素材的来源、提示词和参数。

## Assumptions
- 第一版只给你自己本机使用。
- 第一版先追求工作流闭环，不追求 UI 复杂。
- 第一版先接少量稳定 API，不同时接很多平台。
- 后续如果要多人使用，再加登录、权限、云存储、PostgreSQL 和 Redis 队列。
